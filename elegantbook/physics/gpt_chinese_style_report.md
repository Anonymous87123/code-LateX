## 一、结论摘要

这批聊天记录里的 GPT 中文，最鲜明的味道可以概括为：

> **冷静、克制、行动导向，带有明显“资深工程师边做边汇报”的口吻；习惯先交代动作，再给证据和判断，最后落到下一步。**

它不像日常聊天，也不太像传统客服。更像一个在终端前工作的工程师、技术负责人或项目执行者：不断说明“我先做什么”“当前看到什么”“已经完成什么”“下一步做什么”，并频繁用“先……再……”“不是……而是……”来组织逻辑。

最有辨识度的语言指纹是：

1. **强过程播报**：`我先`、`我会`、`接下来`、`下一步`、`当前`、`目前`。
2. **强完成态表达**：`已经`、`我已经`、`已完成`、`已更新`。
3. **强证据意识**：`确认`、`验证`、`检查`、`结果`、`显示`、`证据`。
4. **强执行倾向**：`直接`、`避免`、`确保`、`落地`、`完整`。
5. **强对照句式**：`不是……而是……`。
6. **强步骤句式**：`先……再……`。
7. **强总结词**：`核心`、`关键`、`结论`、`说明`。
8. **克制的情绪表达**：问号和感叹号都很少，基本不卖萌、不寒暄、不夸张。
9. **固定收尾倾向**：最终答复中经常出现 `下一步`、`建议`、`如果你愿意`。

一句更形象的描述是：

> **像一个可靠但略显程序化的工程搭档：手里拿着检查清单，一边操作，一边报状态，一边收束风险。**

### 2.2 纳入与排除规则

纳入条件：

- 顶层记录类型为 `response_item`；
- `payload.role` 严格等于 `assistant`；
- 内容类型为 `output_text`；
- 正文至少包含一个中文汉字。

排除内容：

- 用户提问；
- system、developer 提示；
- 工具调用和工具输出；
- reasoning、加密推理内容；
- `session_index.jsonl` 等索引文件；
- skills、plugins 目录中的测试夹具；
- 纯英文助手消息。

频率采用“包含该表达的消息数 / 中文助手消息总数”，不是词语总出现次数。因此，一条消息里某个词重复十次，也只计一次。这样更接近“GPT 有多大概率会在一条消息中使用这个说法”。

### 2.3 重要限定

本报告分析的不是脱离环境的“裸 GPT 模型文风”，而是以下因素共同形成的可观察风格：

- GPT/Codex 模型本身的语言偏好；
- Codex 的系统和开发者提示；
- “执行任务时持续汇报进度”的产品交互要求；
- 软件开发、代码审查、测试、文档等任务占比较高；
- 用户偏好的语言、问题类型和上下文；
- 不同日期可能使用了不同模型版本或提示版本。

所以最准确的表述是：

> **这是该 `.codex` 历史记录中，GPT 作为 Codex 工程代理时呈现出的中文风格。**

### 3.3 真正像“口头禅”的表达

从辨识度而不是绝对频率看，最像固定口头禅的是下面这些：

#### `我先……`

这是最强烈的过程型口头禅。它有三个作用：

- 表示已经理解任务，不再等待；
- 宣告第一步动作；
- 给用户一个可预期的执行顺序。

合成句式：

> 我先检查现有结构和相关约束，再确认具体改动范围。

#### `我会……`

这是承诺型表达。通常不是泛泛说“能做”，而是列出马上要执行的动作。

合成句式：

> 我会先补齐测试，再修改实现，最后跑一遍完整验证。

#### `我已经…… / 已经…… / 已完成……`

这是交付型表达，强调工作状态从未完成变为已完成。它让回答很像工单更新或开发日报。

合成句式：

> 已经完成修改，并通过了相关测试。

#### `当前…… / 目前…… / 现在……`

这是状态框定型表达。模型喜欢先声明“在这一刻、在现有证据下”的判断，避免把结论说成永恒事实。

合成句式：

> 当前实现可以工作，但边界条件还没有覆盖完整。

#### `接下来…… / 下一步……`

这是推进型表达。即便当前任务已经完成，GPT 也常把对话继续组织成下一阶段。

合成句式：

> 下一步可以直接补充端到端测试。

#### `直接……`

这是效率型表达，传达“少绕路、给可执行答案”的姿态。

合成句式：

> 这里可以直接复用现有接口，不需要新增一层抽象。

#### `核心…… / 关键……`

这是提炼型表达，用于从大量信息中抓主因。通常会形成“核心是 X”“关键在于 Y”的判断句。

合成句式：

> 关键不在页面样式，而在状态更新没有传到数据层。

#### `如果你愿意……`

这是典型的礼貌式服务收尾。在全部中文消息中覆盖率只有 3.7%，但在最终答复中约为 15%，因此用户会明显感到它反复出现。

合成句式：

> 如果你愿意，我可以继续把剩余模块按同一标准整理完。

它显得友好，但重复使用会产生模板感，也是最容易被识别为 AI 式收尾的表达之一。

## 四、典型句式骨架

### 4.1 “先 A，再 B，最后 C”

覆盖约 19.4% 的中文助手消息，是最典型的任务规划骨架。

常见功能：

- 把复杂任务拆成线性步骤；
- 向用户解释接下来不会失控；
- 让执行过程显得完整、可检查；
- 在一句话里同时表达计划和交付标准。

常见变体：

- 我先看 A，再检查 B。
- 先确认 A，然后处理 B。
- 我会先做 A，再做 B，最后验证 C。
- 先把范围收紧，再决定是否需要改动。

这类句子非常清楚，但用得太多会像流水线播报。

### 4.2 “不是 A，而是 B”

覆盖约 10.3% 的全部中文消息，在最终答复中约 28.6%。这是 GPT 最有辨识度的论证句式之一。

主要用途：

- 纠正表面判断；
- 区分症状和根因；
- 界定两个容易混淆的概念；
- 制造鲜明、易记的结论。

合成句式：

> 这不是渲染层的问题，而是数据在进入组件前已经丢失。

优点是锋利、清楚；缺点是容易把连续谱问题强行写成二元对立。若 A 和 B 实际上同时成立，这个句式会显得过度确定。

### 4.3 “当前状态 + 但是 + 风险/缺口”

典型结构：

> 当前主流程已经可用，但异常路径还没有完整验证。

这种结构体现了工程沟通中的审慎：先确认已有成果，再主动暴露残余风险。它避免只报喜不报忧，也减少“已经完成”被理解成“绝对没有问题”。

### 4.4 “结论 + 证据 + 建议”

最终答复经常采用：

1. 先说结果或结论；
2. 再列出修改、文件、测试或观测证据；
3. 最后给建议或下一步。

合成结构：

> 问题已经修复。原因是 A，改动位于 B，验证 C 已通过。下一步建议补充 D，防止同类回归。

这是典型的技术交付格式，信息密度高，也方便快速扫描。

### 4.5 “为了避免/确保……”

模型经常在动作前补充风险理由：

> 为了避免影响现有行为，我只调整这一处边界判断。

> 我会补充回归测试，确保后续修改不会重新引入问题。

这种句式让方案显得稳健，但也容易形成“每个动作都要附带安全解释”的冗长感。

## 五、语言“味道”的详细拆解

### 5.1 工程师现场口播感

中间进度消息平均只有 83.1 字符，常见内容是：

- 刚看到了什么；
- 下一条命令要查什么；
- 某个假设是否成立；
- 为什么要改变排查方向；
- 还剩下什么验证。

因此整体阅读体验像“结对编程时同伴不断小声汇报思路”，而不是等所有工作做完后一次性给报告。

这种风格的好处：

- 过程透明；
- 用户知道模型没有卡住；
- 便于中途纠偏；
- 每个动作都有上下文。

它的副作用：

- `我先`、`我会`、`现在` 会高频重复；
- 简单任务也可能显得流程很多；
- 用户可能更关心结果，却要连续看到状态播报。

### 5.2 工单、日报和项目管理感

`已经`、`当前`、`下一步` 三组词共同构成了一套近似状态机的叙述：

> 待处理 → 正在检查 → 已确认 → 已完成 → 已验证 → 下一步

这使语言非常适合工程协作，因为它天然回答三个问题：

- 现在在哪一步？
- 已经完成什么？
- 还剩什么？

但在非工程话题里，这种表达可能显得过于项目化、像工单机器人。

### 5.3 证据优先，而不是只给判断

`确认` 覆盖 14.4%，`验证` 覆盖 6.1%，此外还经常出现 `检查`、`结果`、`显示`、`证据`。

典型认知路径是：

> 先观察 → 再确认 → 给结论 → 做验证

模型倾向于把“我认为”写成“我检查后确认”，把“应该没问题”写成“测试结果通过”。这让语气显得可靠、职业化，也降低无依据断言的味道。

### 5.4 强行动性和实用主义

`直接` 的覆盖率达到 24.7%，非常突出。它通常传达以下偏好：

- 能修改就修改；
- 能验证就验证；
- 能复用就不新造抽象；
- 能给具体文件或命令，就不只谈原则；
- 优先完成用户目标，而不是无限讨论可能性。

这是一种明显的“做事型”中文。它比普通说明文更短促、更有动词感。

### 5.5 克制、低情绪、低社交噪声

格式统计显示：

| 特征 | 覆盖消息数 | 覆盖率 |
|---|---:|---:|
| 含冒号 | 62,110 | 52.8% |
| 含项目符号列表 | 18,668 | 15.9% |
| 含代码围栏 | 5,041 | 4.3% |
| 含问号 | 2,618 | 2.2% |
| 含 Markdown 标题 | 2,043 | 1.7% |
| 含感叹号 | 1,227 | 1.0% |

冒号远多于问号和感叹号，说明它更喜欢“标签：内容”“结论：内容”“验证：内容”，不喜欢情绪化感叹或连续反问。

整体很少出现：

- 夸张赞美；
- 网络流行语；
- 拟声词；
- 表情符号；
- 大量寒暄；
- 强烈情绪评价。

所以它给人的感觉是稳、冷静、专业，但有时也略显无趣和机械。

### 5.6 轻度“技术管理黑话”

记录中能观察到一组工程和组织沟通常用词：

- `落地`
- `边界`
- `风险`
- `范围`
- `完整`
- `保持`
- `核心`
- `关键`
- `验证`
- `回归`
- `闭环`
- `上下文`

这些词让语言带有技术负责人或工程经理的味道。它不只是说代码怎么改，也会同时谈影响范围、风险边界和后续验证。

### 5.7 “明确判断”与“保留余地”并存

一方面，模型大量使用：

- `直接`
- `核心`
- `关键`
- `结论`
- `不是……而是……`

这些词让判断显得果断。

另一方面，它也会使用：

- `当前`
- `目前`
- `可能`
- `应该`
- `看起来`
- `大概率`

这些词给结论加上适用范围。

最终形成一种典型口气：

> **在已有证据范围内果断，但为未验证部分保留余地。**

这比一味自信更可靠，但当限定语叠得太多时，会显得啰嗦。

## 六、中间播报与最终答复的差异

### 6.1 中间播报：短、主动、面向动作

`commentary` 共约 8.79 万条，平均 83.1 字符。高频特征包括：

| 表达 | 中间播报覆盖率 |
|---|---:|
| `现在` | 26.0% |
| `我先` | 21.6% |
| `我会` | 19.2% |
| `先……再……` | 18.8% |
| `直接` | 17.4% |
| `确认` | 16.2% |
| `没有` | 14.4% |
| `但` | 11.0% |
| `避免` | 9.0% |
| `当前` | 8.5% |
| `结果` | 8.0% |
| `我已经` | 7.7% |
| `接下来` | 7.3% |

中间播报的典型味道是：

> 我先查 A；现在确认了 B；接下来验证 C。

主语经常是“我”，谓语经常是“看、查、确认、验证、继续、调整”。它关心的是工作进展，而不是完整论证。

### 6.2 最终答复：长、结构化、面向交付

`final_answer` 共约 2.89 万条，平均 1,501.4 字符，是中间播报平均长度的约 18 倍。

在最终答复中，下列表达明显增强：

| 表达 | 最终答复覆盖率 |
|---|---:|
| `现在` | 56.1% |
| `没有` | 48.7% |
| `直接` | 46.7% |
| `可以` | 45.3% |
| `但` | 44.4% |
| `下一步` | 40.0% |
| `当前` | 37.8% |
| `结果` | 35.5% |
| `所以` | 32.4% |
| `建议` | 29.1% |

| `不是……而是……` | 28.6% |

| `我已经` | 21.4% |
| `先……再……` | 21.4% |

需要注意：`现在`、`没有`、`但`、`可以` 都是宽泛常用词，可能出现在代码解释、否定条件或引用中，不能完全视为口头禅。不过它们的高覆盖率仍说明最终回答喜欢描述当前状态、明确能力边界，并用转折补充限制。

最终答复的典型组织是：

> 已完成什么 → 改了什么 → 为什么这样改 → 如何验证 → 还有什么限制 → 下一步建议

它更像交付报告，而不是聊天回应。

## 七、这种文风的优点

### 7.1 可预期

`我先`、`接下来`、`下一步` 让用户知道任务会如何推进，不容易产生“模型是不是卡住了”的不确定感。

### 7.2 可验证

强调 `确认`、`验证`、`结果`，能把判断和证据绑定起来。对于代码、配置、构建和测试任务，这一点非常实用。

### 7.3 责任主体清楚

大量使用 `我会`、`我已经`，明确哪些动作由模型执行，哪些建议留给用户。这比模糊的被动句更适合协作。

### 7.4 结论容易扫描

`核心`、`关键`、`结论`、冒号和列表，让长回答可以快速定位重点。

### 7.5 风险意识强

`避免`、`确保`、`边界`、`风险`、`回归` 等词表明模型会主动考虑副作用，而不只追求眼前能跑。

### 7.6 语气稳健

低感叹号、低反问、少寒暄，使语言在工程环境里显得专业，不会因为过度热情干扰信息传递。

## 八、这种文风的缺点和 AI 痕迹

### 8.1 过程播报过密

当每一步都以“我先……”“接下来……”开头时，用户会感到模型在重复描述工作，而不是推进工作。尤其对简单任务，播报成本可能超过任务本身。

### 8.2 完成态词语重复

`已经` 出现在 37.9% 的消息中。它能说明进展，但连续出现“已经确认、已经完成、已经验证”时，容易像自动生成的状态报告。

### 8.3 “直接”使用过多

`直接` 覆盖 24.7%。它带来干脆感，也可能造成三种问题：

- 把复杂取舍说得过于简单；
- 让建议显得武断；
- 多次出现后失去强调效果。

### 8.4 “不是……而是……”容易制造假二分

这个句式在最终答复中覆盖约 28.6%，频率很高。很多技术问题并非 A 与 B 二选一，而是 A、B 按不同权重共同作用。过度使用会让论证显得聪明但不够精确。

### 8.5 “如果你愿意”模板感明显

它本来是礼貌表达，但最终回答约每 6 至 7 条就有一条包含这句话。频率过高时，会像固定客服尾巴，而不是根据上下文自然收尾。

### 8.6 结构词太多

`当前`、`接下来`、`下一步`、`核心`、`关键`、`结论` 都是很强的路标。路标过密时，文章会像演示文稿或周报，缺少自然连贯的叙述。

### 8.7 技术管理词会造成距离感

`落地`、`边界`、`范围`、`闭环`、`风险` 等词在工程场景准确，但如果主题是生活、写作或情绪沟通，就会显得不合时宜。

### 8.8 第一人称偏多

`我先`、`我会`、`我已经` 有助于明确责任，却也会让文本过度围绕模型自身的动作。用户真正关心的往往是结果，因此最终答复中可以更多使用结果主语。

例如：

- 模板感较强：`我已经修改了配置，并且我已经完成了验证。`
- 更自然：`配置已修改，相关验证通过。`

## 九、最能复刻这种风格的句型模板

以下均为根据统计特征编写的合成模板，不是聊天原文。

### 9.1 开始执行

> 我先检查 `{对象 A}` 和 `{对象 B}`，确认 `{关键约束}` 后再决定具体改法。

> 我会先梳理现有实现，再做最小范围修改，最后补一轮验证。

### 9.2 汇报发现

> 当前可以确认 `{现象}`，但 `{边界/风险}` 还需要进一步检查。

> 现在已经定位到 `{位置}`，问题不是 `{表面原因}`，而是 `{根本原因}`。

### 9.3 解释判断

> 核心在于 `{核心机制}`；`{次要现象}` 只是它在当前路径上的表现。

> 关键不是增加更多逻辑，而是让 `{状态/数据/约束}` 在现有边界内保持一致。

### 9.4 说明修改

> 已完成 `{改动}`。这次只调整 `{范围}`，避免影响 `{既有行为}`。

> 我直接复用了 `{现有能力}`，没有新增不必要的抽象。

### 9.5 交付验证

> 修改已经完成，`{测试/构建/检查}` 通过。当前没有发现新的回归。

> 验证结果说明 `{结论}`，剩余风险主要集中在 `{未覆盖边界}`。

### 9.6 给下一步

> 下一步建议补充 `{动作}`，这样可以把 `{风险}` 一并覆盖。

> 如果你愿意，我可以继续处理 `{自然延伸任务}`。

## 十、如何写得像这种 GPT 中文

如果目标是主动复刻这种语言风格，可以遵循以下规则：

1. 开头不寒暄，直接说动作、状态或结论。
2. 复杂任务用“先 A，再 B，最后 C”。
3. 每个判断尽量附带证据、文件、测试或观测结果。
4. 用“当前/目前”限定结论范围。
5. 用“核心/关键”提炼一层，但不要同时重复使用。
6. 用“不是 A，而是 B”纠正误区，但只在确实互斥时使用。
7. 修改说明要交代影响范围和风险控制。
8. 最终答复按“结果、改动、验证、限制、下一步”排列。
9. 少用感叹号、反问、夸张修饰和长寒暄。
10. 句子以动词为中心：检查、确认、修改、补充、验证、完成。

一个简化公式是：

> **动作承诺 + 当前发现 + 核心判断 + 实际改动 + 验证证据 + 下一步**

## 十一、如何减少这种 GPT/AI 味

如果希望保留专业度，但让文字更像自然的人类工程师，可以这样调整：

1. 每次答复最多使用一次 `我先` 或 `我会`。
2. 已经明确完成时，省略重复的过程叙述，直接给结果。
3. 将一半以上的 `直接` 删除，只有在确实存在绕路方案时才保留。
4. 将 `不是 A，而是 B` 改成更精确的权重表达，例如“主要原因是 B，A 也有次要影响”。
5. 不要固定以 `如果你愿意` 结尾；有自然下一步时再提出。
6. 避免在同一段同时使用 `核心`、`关键`、`本质上`。
7. 少写“为了确保/为了避免”的目的从句，把理由压缩到必要位置。
8. 用具体事实替代抽象管理词，例如用“这三个测试未覆盖”替代“测试闭环不完整”。
9. 简单问题用一段话解决，不强行加标题、摘要和下一步。
10. 让句长有变化，不要每段都采用相同的三段式。

改写示例：

> AI 味较重：我已经完成了相关修改。核心问题不是配置缺失，而是状态没有正确同步。下一步建议补充测试，以确保后续不会出现回归。如果你愿意，我可以继续处理。

> 更自然：修改完成，问题出在状态同步，不是配置缺失。相关测试已通过；再补一个异常路径用例，就能覆盖这次回归。

## 十二、综合风格画像

可以从几个维度给这批 GPT 中文做一个定性画像：

| 维度 | 强度 | 说明 |
|---|---|---|
| 行动导向 | 很强 | 经常立即说明要做什么 |
| 过程透明 | 很强 | 大量中间进度播报 |
| 证据意识 | 强 | 高频确认、检查、验证、结果 |
| 结构化 | 强 | 冒号、列表、阶段词明显 |
| 结论性 | 强 | 喜欢核心、关键、不是而是 |
| 风险意识 | 强 | 强调避免、确保、边界和回归 |
| 情绪表达 | 很弱 | 感叹、玩笑和情绪词很少 |
| 寒暄程度 | 很弱 | 通常直接进入任务 |
| 口语自然度 | 中等偏低 | 清楚，但有模板化和工单感 |
| 专业可信感 | 较强 | 像技术负责人或资深工程师 |
| AI 可识别度 | 中等偏高 | 固定路标词和收尾方式明显 |

最终可浓缩成六个关键词：

> **主动、克制、结构化、证据化、工程化、略模板化。**

或者用一句更有“味道”的话概括：

> **它说中文时像在写一份实时更新的技术工单：先报动作，再报状态；先收紧范围，再确认根因；做完一定说验证，结尾经常还要留一个下一步。**

## 十三、统计解读注意事项

1. “包含率”不等于词频；一条很长的最终答复只计一条。
2. `现在`、`没有`、`但`、`可以` 等宽泛词可能受到代码解释和引用内容影响。
3. Markdown 代码块、命令和文件名没有从正文中逐段剔除，因此部分技术词频可能略高。
4. 当前会话也位于 sessions 目录中；两次扫描之间新增了极少量消息，导致第二轮分阶段统计比第一轮多 3 条，对百分比没有实质影响。
5. 归档会话与当前会话按文件位置一并扫描；未对语义近似回答做去重。
6. 子代理、不同任务类型和不同模型版本可能混在总体中，因此这里描述的是整体“中心风格”，不是每条回答都具备全部特征。
7. 软件工程任务天然会提高“验证、测试、风险、边界、下一步”等词的频率。
8. 如果要比较不同模型，需要进一步按 `turn_context.model`、日期和任务类型分组，才能避免样本构成差异造成误判。

## 十四、v19 对去 AI 文风 Skill 的实证补充

三组真实 GPT TeX 前向测试暴露了一个重要区别：旧 validator 能让三个候选全部 `PASS/0`，但盲评
只有课程稿合格；建模稿删掉了表格职责，研究稿还新增病句。也就是说，“禁词减少、公式没变、句子
更短”并不等于文字更自然。

修复后，Skill 增加段落职责、论证层级、成对优越性和焦点作用域四道门，并明确拦截“实证闭环”与
“抽象关系由……出发，经……最终落到……”等落地句壳。fresh 盲评最终三组都偏好改后：课程
8.8、建模 8.8、研究 8.7，忠实度均不低于 9.7。`SKILL.md` 同时由 443 行压到 280 行。

这仍不是总体生成资格通过：487 项测试和两次一致投影只证明工具链可复现，资格审计仍是
`NOT_EVALUATED / 0 of 179`。完整证据见
[v19 章节](pure_academic_humanize_skill_report.md#23-v19用真实成对质量替代删词即完成)。

## 十五、v20 对“少改”和定点改写的实证补充

新增四类真实 MD/TeX 测试后，最重要的新结论不是又收录了多少 AI 词，而是三种更具体的失效方式：

1. `PASS/0` 的 GENERAL 改写仍可能比原文差。把“造成论证前置过多”改成“使过多论证被前置”，
   正是典型的“去 AI 时反而写出 AI 味”；盲评为改后 8.3、原文 8.7。
2. supplied facts 中同一公式只出现一次，不代表结果和讨论只能引用一次。旧 occurrence budget 把
   `R_S=0` 的正常复用误报为补造；现在同值复用进入 `REVIEW/2`，新值仍 `FAIL/1`。
3. 检测报告的标红片段映射正确，也不能证明代理只改了该片段。新 `--report-scope` 把 source SHA、
   UNIQUE 字符区间和范围外文本绑定到统一验证器；旧版假 PASS 的越界稿现被硬拦截。

post-fix fresh 代理对原本 8.7 的 GENERAL 段落逐字 `NO_CHANGE`，对高质量课程段也再次不改；报告
改写盲评 9.1，高于原文 8.7；DRAFT 逐句来源正确、内容可交付，但自动状态没有冒充 PASS。完整证据见
[v20 章节](pure_academic_humanize_skill_report.md#24-v20general-克制材料约束起草与报告范围闭环)。

## 十六、v21 对“结构型 AI 味”的实证补充

此前报告已经指出 GPT 正式稿常把一个自然论证拆成若干职责近似的小段，并给每段补齐定义、解释和
收尾。v21 证明，解决这类结构味不能只写“合并重复段落”规则：旧长文入口实际上把全部 unit 固定为
`BALANCED`，并不存在可执行的 STRUCTURAL 管线。

现在每个结构候选都要提交来源段 plan，绑定段 ID、SHA-256、原顺序、职责、保护项归属和目标段。
只允许 unit 内、标题锁定、相邻同职责段合并；公式和数字只能随来源段整体移动。验证器先生成只重排
原段的结构基线，因此既不会因公式顺序变化误杀合法整段移动，也不会放过公式离段或数值改写。

四份真实长文的 v21 prepare/inventory 显示，自动可移动面远小于表面上的“段落很多”：`physics1.tex` 为 51/507，微信
`main.tex` 为 4/65，详细风格报告为 7/114，Skill 长报告为 73/436。公式密集稿和表格/代码密集报告
尤其保守。这里的分母只统计 PENDING unit 的来源段，不是全文改写成功率；四个工件当时都只执行
prepare，并未完成全文 STRUCTURAL。这说明结构过度组织是可观察的 GPT 味，但不能据此授权任意章节重排。

独立子代理 fresh trial 合并了 `physics1.tex` 一个题解的两个相邻段：plan、公式、TeX、数字和高风险词项门均
通过；然而删去重复收尾使 `必须` 从 3 次降为 2 次，触发模态范围 `REVIEW/2`。候选没有发布，结构
语义保持 `NOT_EVALUATED`。该试验没有 sealed prompt、runner receipt 或外部隔离证明，不能称资格级
盲测。这是正确的拒绝：机械结构映射通过，不等于删句后的语义强度已经得到证明。

本轮共运行 504 项，整体 `OK (skipped=1)`；两次 30 文件投影逐文件一致；生成资格仍为
`NOT_EVALUATED / 0 of 179`。详见
[v21 章节](pure_academic_humanize_skill_report.md#25-v21structural-从名义强度变成来源段可追溯管线) 与
[独立子代理试验](build/maturity-v21-20260717/postfix-structural-blind/evaluation.md)。

## 十七、v22 对“完成态 AI 味”的工程补充

AI 味不只出现在词句里，也出现在工具对自身结果的包装方式上。v21 可能同时给出顶层
`status=PASS` 和 `structural_semantic_mapping=NOT_EVALUATED`：字段逐个看都能解释，但整体仍像典型的
“先宣告完成，再在后文附限制”。v22 把这种完成态冲突直接从状态机里删掉。

现在，结构候选可以 `candidate_assembly_status=PASS`，但只要真实移动后的语义尚未外部复核，最终
`delivery_gate_status/status` 就必须是 `REVIEW/2`，候选只能写入 `rendered_review/`。同时生成绑定
before/baseline/after、plan、上下文、warning 和工具策略 hash 的结构语义 review request；模型自己的
reason、调用方 HUMAN 标签和本地自填 clearance 都不能把它升级为 PASS。

真实 `physics1.tex` 控制链提交了 50 个可编辑 unit：49 个 `NO_CHANGE`，1 个结构合并。完整候选的
assembly 为 PASS，但 delivery 为 REVIEW，`humanize_completion_claim_allowed=false`；另一条保留
`必须: 3 -> 2` warning 的路径则有 1 个 UNRESOLVED，partial 与源文件逐字节相同。两条路径的编译和
fresh second pass 都是 `NOT_RUN`。完整候选的实质变化只是删除两个题解段之间的空行，因此它证明
的是“候选和正式交付不再混淆”，不是“全文已经去掉 AI 味”。

当前共运行 507 项测试，整体 `OK (skipped=1)`；两次 30 文件 projection 一致；资格矩阵为

## 十八、v23 对“结构型 AI 味”跨分块处理的补充

“题解或论证被切成多个职责近似的小段”是本批 GPT TeX/Markdown 中可观察的结构味，但它不能直接
转化为任意跨段权限。v23 只开放严格的 `ADJACENT_PAIR`：两个 unit 必须同文件、同标题、物理相邻，
且 scene 与 Voice 一致；候选 inventory 不是执行请求，还要提交绑定 transaction bundle。

执行时，两个 fragment、联合来源段覆盖、DOCUMENT gate 和全文重复门共同构成原子事务。任一侧
失败就整对回滚；成功也只写：

```text
candidate_assembly_status=PASS
delivery_gate_status=REVIEW
status=REVIEW
exit_code=2
structural_semantic_mapping=NOT_EVALUATED
```

真实压力结果很保守。默认 `7000/1200` 字符预算下，`physics1.tex`、微信 `main.tex` 和 GPT 风格报告
均为 0 个 pair；改为 `1200/0` 后，physics 为 67 units、57 PENDING、7 个 pair，另外两份仍为 0。
这说明工具不会为了产生结果而跨标题造候选。

真实 smoke 只从 `physics1.tex` 冻结快照提取 9,462 字节局部片段，把一个完整段跨相邻 unit 搬运，
不改段内字节。两个 fragment、DOCUMENT 与 atomic gate 均 PASS，但最终仍为 `REVIEW/2`，只生成
`rendered_review/` 和外部语义复核请求；compile 为 `NOT_RUN`，原 `physics1.tex` 未修改。这证明
原子状态机可用，不证明全文文风改善或物理内容正确。

隔离前向代理还给出一个更贴近实际写作的结果：同一局部稿虽然形成 1 个合法 pair 候选，但移动会
打乱两道物理题的题干-解答对应，因此代理提交两个 `NO_CHANGE`，PATCH 为空，源与派生稿 SHA
相同；微信稿保持 `EMPTY/0`；显式跨三个 unit 并要求标完成的请求被 `UNRESOLVED/NOT_RUN` 拒绝。
这说明“检测到结构味”没有被实现成强制改动配额。

当前回归为 540 项测试，整体 `OK (skipped=2)`；20 个脚本 compile/help 通过；两份 30 文件
projection 逐字节一致；生成资格仍为 `0 PASS / 0 FAIL / 184 NOT_EVALUATED / exit 2`。不支持超过
两个 unit、非相邻、跨文件、跨标题、标题解锁、拆段或整段删除。详见
[v23 详细章节](pure_academic_humanize_skill_report.md#27-v23相邻双-unit-structural-原子事务) 与

## 十九、v24 对“两个 NO_CHANGE 不等于候选已审”的修正

v23 的隔离 physics 前向有 2 个 unit 和 1 个 READY `STX-*` 候选。代理判断跨块移动会打乱题干与
题解，于是提交两个普通 `NO_CHANGE`；旧 finalizer 返回 `PASS/0`、`FINAL` 和正式 `rendered/`。
当时报告据此写成“候选不适合，所以没有强行改动”。语言判断本身可能合理，但机器证据并没有闭合
候选：`NO_CHANGE` 只回答各 unit 正文是否改，不能回答某条 transaction 边是否执行或拒绝。

v24 为每个冻结候选增加独立 disposition：

```text
EXECUTED | DECLINED | PENDING
```

execution 必须绑定双 fragment；decline 必须绑定 transaction/inventory、冻结顺序的两个 chunk/Voice、
至少 8 个汉字的具体理由和两侧来源段证据。空泛“已审阅、无需调整、结构合理”、stale hash、单侧或
重复 evidence、execution+decline 冲突都不能占据 disposition。任一候选 `PENDING` 时，即使两个
unit 都是 `NO_CHANGE`，仍固定 `REVIEW/2`、覆盖声明 false，并禁止正式 `rendered/`。

真实 v24 前向重新处理同一劈尖局部稿：2 个 PENDING unit、1 个 READY candidate、119 个保护跨度。
代理提交 1 个 `NO_CHANGE`、1 个局部 `REWRITE`，并以
`QUESTION_ANSWER_PAIRING_RISK` 明确 decline 该 pair：左侧末段是题干，右侧开头段是对应题解，跨
unit 移动会打乱顺序。最终候选 `total/executed/declined/pending=1/0/1/0`，这是 **v24 pre-fix 历史
`PASS/0`**，不再是现行交付状态；v25 同范围要求 paired-quality request，当前顶层为 `REVIEW/2`。
实际结构变化 0；compile 与 fresh second pass 均未运行，全文 Humanize 声明仍为 false。

局部改写删除了“看上去……其实真正……也就是说”的主持式壳，把“这里空气劈尖取”改成直接陈述，
把“这里不能把 30 条条纹……”改成范围更具体的“计算时不能把……”。独立盲读轻微偏好改后：
自然度 `8.8 -> 9.0`，清晰度 `9.1 -> 9.2`，信息忠实度均为 `10.0`；但“但求解先要确定”仍略显
紧缩，宜写成“但求解时须先确定”。这项盲读只比较语言和内部信息保持，不是物理正确性验证。

该历史章节当时共运行 549 项测试；v25 当时覆盖为 572 项（570 执行通过、2 环境跳过），资格矩阵为
`0 PASS / 0 FAIL / 188 NOT_EVALUATED / exit 2`。两份 30 文件 projection 逐字节一致，并剥离资格控制面。
这些结果证明候选完成态
更可信，不证明生成模型已经成熟或全文一次改写稳定合格。
