## 1. 场景声音

建模与工程论文的自然感来自务实取舍：作者清楚哪些设计决定值得解释，哪些设置只需交代，哪些结果真正改变判断。目标不是写得随意，而是避免每个子问题都像按同一竞赛模板交付。

理想特征：

- 模型、变量、情景、方案和结果先于写作路线出现；
- 关键设计取舍详写，例行设置和常规运算简写；
- 子问题结构随任务变化，不复制“分析—优点—局限—补强”；
- 结论有工程判断，但不使用项目经理命令；
- 摘要突出共享主干和关键输出，不逐问报到；
- 同一结论只完整出现一次，其他位置短回指；
- 允许局部停止，不给每节自动追加未来工作。

本 Skill 不判断模型、数据、参数、公式、实验或结论是否可靠。它只调节表达权重和读感。

### 1.1 规则前置裁决

只在上下文确认管理腔、模板复用或解释冗余后启动规则。正式模型术语、情景名称、等权因素、定义/假设/结果报告和用户锁定结构优先。`框架、机制、角色、因此、结果表明` 等词本身不是病灶；冲突时使用 `KEEP/REVIEW/NO_CHANGE`。

建模场景的独有目标不是通用“对象先行”，而是让作者取舍落到可读的模型操作、比较对象和方案后果。不得补造原文没有的工程建议。

## 2. AI 文风黑名单

### 2.1 编辑指令泄漏

典型形式：

```text
正文里既要……也要……
参数扫描的写法要服务于……
表格的作用在于让读者……
如果把它放到图上……
对论文写作而言，本问更适合……
```

这些句子在告诉作者怎么写，而不是向读者陈述模型或结果。改成正文已经做出的具体动作。

### 2.2 五问同构

真实建模稿常出现：

```text
本问的优点不在于 A，而在于 B
其局限同样需要写清
后续若继续补强，应优先 C
```

五个问题反复复制这三个槽位，会形成最强机器模板。各问应按对象差异组织，共同限制或后续动作集中一次。

### 2.3 工程管理声线

```text
钉死 / 收口 / 落地 / 主线 / 接口 / 预埋
门禁 / 驾驶舱 / 闭环 / 必须写 / 不要写
```

这些词可以存在于内部施工图，不进入论文正文。改成模型对象、算法动作或具体比较。

### 2.4 完整链条迷恋

“上游—中间—下游”“参数点—代表轨道—尾窗特征”“结构—校准—验证—决策”若不断被命名，会让链条标签比内容更显眼。只有后文反复调用时才命名。

### 2.5 抽象工程名词堆积

```text
框架 / 模块 / 口径 / 角色 / 链条 / 路径
机制 / 维度 / 抓手 / 闭环 / 体系
```

连续出现时，句子会失去具体主语。至少把一个抽象名词还原成变量、模型、步骤、图表或动作。

### 2.6 摘要与结尾流水账

- 摘要按“首先、其次、再次、最后、针对问题一”逐问列模型；
- 结论逐章回放所有结果；
- 每节自动补局限和未来补强；
- 最后再升到治理价值或完整方案。

## 3. 建模纯文风规则

### A. 去除编辑元话语

`MOD-HUM-01 MUST`：删除所有面向作者的编辑指令。把“正文应如何写”改成正文已完成的具体陈述。

`MOD-HUM-02 MUST`：删除“表格的作用是、图上应当、对论文写作而言”。直接指出表或图中的对象与变化。

`MOD-HUM-03 MUST`：内部计划中的“主线、钉死、收口、接口、预埋、门禁、落地”不得迁入正式正文。

`MOD-HUM-04 SHOULD`：章节开头不复述标题或交代“本节将做什么”，直接进入当前模型、变量变化、情景或判断。

### B. 打破子问题模板

`MOD-HUM-05 MUST`：同一节不得复制固定“优点—局限—未来工作”槽位。按本节真正重要的信息选择结构。

`MOD-HUM-06 MUST`：连续两个子问题不得使用同一句首骨架；第三次复用“本问关注的不是……而是……”必须改写结构。

`MOD-HUM-07 MUST`：五个子问题不得拥有近似相同的段落数、句数和收尾方式。篇幅随模型职责变化。

`MOD-HUM-08 MUST`：共同的限制、说明或后续方向集中出现一次，不在每问末尾复制。

`MOD-HUM-09 MUST`：每段只承担一个主要写作动作。解释、评价、限定、建议和展望不必在同段齐全。

`MOD-HUM-10 SHOULD`：不同问题若共享上游对象，用回指连接；不要为了每问自足重新铺背景。

### C. 句首与衔接

`MOD-HUM-11 MUST`：“不是 A，而是 B”只保留真实互斥纠偏。若 B 可独立成立，直接写 B。

`MOD-HUM-12 MUST`：不用“因此、进一步、与此同时、也正因如此”维持表面连续。优先由变量、图表、参数或上文对象承接。

`MOD-HUM-13 MUST`：“需要指出的是、值得注意的是”通常删除。重点通过句位、篇幅和具体名词体现。

`MOD-HUM-14 SHOULD`：结果段不强制以“结果表明”起句。可直接从数值变化、排序变化、曲线转折或方案差异起笔。

`MOD-HUM-15 SHOULD`：段落主语优先使用模型、变量、算法步骤、情景或图表，减少“本文、该框架、这一机制、这种口径”。

### D. 解释密度与抽象名词

`MOD-HUM-16 MUST`：公式后的文字只说明本式在当前叙述中的关键作用，不逐项复述符号已经表达的内容。

`MOD-HUM-17 MUST`：已由公式、符号、图注或表头清楚表达的信息允许省略，不换词再说一遍。

`MOD-HUM-18 SHOULD`：关键模型选择或关键结果允许长段展开；常规设置、例行代数和次要背景应明显缩短。

`MOD-HUM-19 MUST`：三个以上“框架/机制/链条/口径/角色”若只是抽象评价，改回具体对象或动作；正式术语、并列分类、引号内名称和后文复用名词使用 `KEEP/REVIEW`，不按词数强制改写。

`MOD-HUM-20 MUST`：“这样做的好处是”改为该设计造成的直接变化，不先评价再解释。

`MOD-HUM-21 SHOULD`：链条标签只有后文会反复调用时才命名。一次性关系直接写进句子。

### E. 摘要、结果与结论

`MOD-HUM-22 MUST`：摘要不逐问报到。先交代共享模型主干，再突出最重要取舍和少量核心输出；次要子问题移入正文。

`MOD-HUM-23 MUST`：结果段不复刻问题分析的顺序说明。按真正出现差异的结果组织，不按完成清单组织。

`MOD-HUM-24 MUST`：同一结论首次完整表达后，后文短回指，不在分析、结果、优缺点和结论中反复展开。

`MOD-HUM-25 MUST`：结论不复述全部章节。优先留下一个中心判断和至多一个真正必要的后续动作。

`MOD-HUM-26 MUST`：未来工作集中出现一次，不在各小节自动追加“后续若继续补强”。

`MOD-HUM-27 SHOULD`：治理建议或工程建议可以直接，但不在最后强制升华到宏大价值。

### F. 节奏与机器完美感

`MOD-HUM-28 MUST`：不为凑整齐制造三条、四条或五条并列项。能合并成一句的项目直接合并。

`MOD-HUM-29 MUST`：一句超过两个转折或递进层级时，优先在对象变化处拆句；不得按逗号平均截断。

`MOD-HUM-30 SHOULD`：连续长句之间允许一个承载具体判断的短句，不能只写“这一点十分重要”。

`MOD-HUM-31 MUST`：工程取舍可以明确，但不把选择写成“唯一、最终、必须”的项目命令；改成当前文本已经采用的动作。

`MOD-HUM-32 MUST`：适度不完美指信息取舍、篇幅坡度、局部省略和不对称结构，不指制造错误、随意口语或表达毛刺。

`MOD-HUM-33 MUST`：先删模板句，再调段落，再改词。只把“综上”换成“总而言之”不算 Humanize。

`MOD-HUM-34 MUST`：文风终审只回答“是否像统一模板生成”，不得评价模型、公式、数据或实验。

`MOD-HUM-35 MUST`：关键建模取舍应连接到原文已有的操作变化，例如改变哪个项、固定哪个对象或比较哪组情景；不要只写“更合理、更系统”。

`MOD-HUM-36 MUST`：结果解释优先说明原文已有的比较如何影响方案选择或工程阅读，不停留在通用“观察先行”；原文没有决策后果时不得补造。

`MOD-HUM-37 SHOULD`：假设、操作、输出和选择四者不必组成固定四格。只保留当前段落实际发生的关系，并维护假设/结果等言语状态。

`MOD-HUM-38 MUST`：若改写动作在 `COURSE` 或 `RESEARCH` 下完全相同，检查是否只应用了通用内核；建模专属改写至少体现一个操作、情景比较或方案后果，否则使用 `GENERAL`。

## 4. 节奏调控

### 4.1 建立篇幅坡度

一节中的篇幅应呈坡度，而非平地：

```text
关键设计取舍 / 决定性结果      最长
必要模型定义 / 比较说明         中等
例行设置 / 常规计算 / 过渡      最短
```

若五个问题篇幅完全相同，先决定哪一问建立共享模型、哪一问只新增约束、哪一问只是情景比较，然后重新分配。

### 4.2 建立句长波形

- 对象和结果用短句落下；
- 复杂设计理由可用长句展开；
- 一个段落中只保留一个主要长句中心；
- 句长变化服从对象变化，不为了指标好看随机拆分。

### 4.3 建立解释密度差

首次定义共享对象时完整说明；后续子问题只解释新增部分。图表中一眼可见的趋势不逐条复述，异常、反转或真正改变判断的位置才展开。

### 4.4 用对象完成衔接

上一段以参数 (K) 结束，下一段可以直接从 (K) 的扫描范围起笔。上一段给出情景排序，下一段直接讨论发生交换的位置。无需“进一步而言”。

### 4.5 允许局部停止

小节在结果或取舍落地后可以结束，不自动追加评价、限制和未来工作。章节间已有标题时，不必另造承上启下句。

## 5. 机器完美感破除

合理方式：

- 问题一因建立共享模型而详写，问题二只讲新增约束；
- 结果段只展开一个发生反转的情景，其他情景合并说明；
- 某节没有“优点”，某节没有单独“局限”；
- 摘要只保留两个最有区分度的输出；
- 结论不逐问回收；
- 常规公式只回指，不逐行讲解；
- 一个具体结果后直接结束小节。

禁止方式：

- 故意把工程文字改成聊天；
- 随机删掉参数、单位或术语；
- 添加“我觉得、可能吧”等伪犹豫；
- 用错字、病句和碎句制造不完美；
- 把所有正式词替换为口语词。

## 6. 章节差异

| 章节 | 自然重心 | 可压缩内容 | 最危险 AI 模板 |
|---|---|---|---|
| 摘要 | 共享模型主干、关键取舍、少量输出 | 逐问流程、全部数值、自动建议 | 首先/其次/再次/最后报到 |
| 问题分析 | 每问真正不同的难点 | 写作提示、图表用途、未来补强 | “不是 A 而是 B”五次复用 |
| 模型建立 | 对象和关键设计选择 | 公式字幕、抽象框架评价 | 每式定义—解释—好处—承接 |
| 求解/算法 | 关键动作和变化点 | 例行步骤、操作播报 | 先/再/然后/最后流水线 |
| 结果 | 最值得解释的差异 | 可见趋势、方法路线复述 | 每图一套“结果表明” |
| 讨论 | 一个主要取舍或影响 | 对称优缺点、每问限制 | 优点—局限—补强三格 |
| 结论 | 中心判断、必要动作 | 全文目录、全部结果复述 | 验收清单 + 治理升华 |

## 7. 终审问题

1. 正文是否仍出现“正文要写、表格作用、如果放图上”等编辑提示？
2. 五个子问题是否仍使用相同句首、段数和收尾？
3. “优点—局限—补强”是否被复制？
4. 摘要是否仍逐问报到？
5. 抽象工程词是否压过模型、变量和结果？
6. 公式与图表是否仍被逐字复述？
7. 重点问题是否获得明显更多篇幅？
8. 结论是否只保留真正重要的判断，而非章节验收？
9. 是否存在自然停止、短回指和不对称结构？
10. 是否完全没有输出模型精度、数据、实验或正确性判断？
