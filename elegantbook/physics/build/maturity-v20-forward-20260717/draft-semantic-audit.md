# DRAFT 语义与交付审计

## 审计范围与总判定

本审计仅依据 `cases/draft/facts.tex`、`cases/draft/output.tex` 和 `cases/draft/validation.json`。逐句核对后，`output.tex` 的全部事实陈述均可由 `facts.tex` 明确蕴含，未发现新增数字、参数、机制、来源、验证、引文或实质性范围扩张。文本严格由两个自然段组成，中间以空行分隔，无标题、无列表，符合指定的“问题二结果与讨论”体例。语义、结构和文体层面可直接交付。

`validation.json` 的 `FAIL` 属于**确定性检查误报**，不是稿件真实越界，也不是“真实越界与误报二者都有”。它确实造成当前自动门禁不能标记为通过，但不构成稿件语义不受来源支持的证据。

## `validation.json` 的 FAIL 归因

门禁失败由唯一一条硬错误触发：`invariants.errors[0].code` 为 `DRAFT_MATH_NOT_SUPPLIED`，其 `details.not_supplied` 只列出 `\(R_S=0\)`；同一对象的 `message` 是 “Math span in draft is absent from supplied content”。然而，`facts.tex` 第 10 行明确包含完全相同的数学片段 `\(R_S=0\)`，所以“absent from supplied content”在语义和字面存在性上都不成立。

JSON 中的计数揭示了误报来源：`draft_surface_source_check.details.DRAFT_MATH_NOT_SUPPLIED` 给出 `draft_count: 6`、`supplied_count: 5`，而稿件在第一段结果报告和第二段机制解释中各使用一次 `\(R_S=0\)`；事实材料使用一次。检查器显然按出现次数消耗来源片段，把第二次复述当作“未供应”，而不是判断该数学表达是否存在于事实材料。重复使用一个已供应的参数设定来先报告结果、再解释对照用途，不会生成新参数或新事实，因此这是确定性表面检查的假阳性。

其他字段也不支持“真实越界”的结论：

- `DRAFT_NUMBER_OR_UNIT_NOT_SUPPLIED.not_supplied` 为空，说明没有被检出的新增数字或单位；
- 归因、代码、关键命令、形式陈述、乱码、受保护术语和直接引文各项的 `not_supplied` 均为空；
- `introduced_findings`、`pending_warnings`、`unaccepted_warnings` 与 `unexplained_high_findings` 均为空；
- `style_signal_layer_status` 和 `speech_act_layer_status` 均为 `PASS`；
- `semantic_source_check` 为 `NOT_EVALUATED`，`academic_correctness` 也为 `NOT_EVALUATED`，`review_reasons` 仅为 `semantic_source_not_evaluated`。因此 `semantic_entailment_certified: false` 表示检查器没有完成语义认证，不能解读为已经认定语义不蕴含。

最终归类：**确定性检查误报**。`delivery_gate_status: FAIL`、`hard_invariant_layer_status: FAIL`、`exit_code: 1` 是该误报造成的真实流程状态；它们意味着当前文件无法在不处理检查器计数规则的情况下取得自动门禁通过，但不改变稿件本身可由事实材料逐句支持、内容层面可直接交付的判断。
